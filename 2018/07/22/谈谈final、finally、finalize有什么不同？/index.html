谈谈final、finally、finalize有什么不同？

基础知识
  • Java核心技术36讲
  • 基础知识

final

从语法和使用实践角度：

final可以用来修饰类、方法、变量，修饰class代表不可以继承扩展，修饰变量是不可以修改的，修改方法是不可以被重写的。(override)

推荐使用final关键字来明确表示代码的含义和逻辑意图：

  • 将方法或者类声明为final，明确这些行为是不许修改。
  • 使用final修饰参数或者遍历，可以清楚地避免意外赋值导致的编程错误。
  • final变量产生了某种程度的不可变(immutable)的效果，用于保护只读数据。
    • 在并发编程中，明确不能赋值final变量，有利于减少额外的同步开销以及省去一些防御性拷贝(copy-on-write)的必要。
    • 匿名内部类，访问局部变量时，局部变量为啥要用final来修饰吗？因为Java inner class实际会copy一份，不是去直接使用局部变量，final可以防止出现数据一致性问题。

特别提一句，《java编程思想》中讲到final方法时提到，使用final方法原因有两个：

  1. 锁定方法。防止任何继承类修改、覆盖 。
  2. 效率。在java早期实现中，如果将一个方法指明为final，就是同意编译器将针对该方法的调用都转化为内嵌(内联)调用……

final与内联

      按照理解，虚拟机执行正常方法调用的步骤：参数压栈–>跳转方法处执行–>调回–>处理栈参数–>处理返回值。相反，方法内联，是直接将方法展开，将方法的调用替换成方法体本身(实际代码)。作用在于减少了方法调用开销，提升效率，但是会占用更多的内存空间(内联方法调用多则出现方法体本身的地方就多)。比如：

实现immutable的类，需要满足下列条件：

  • 将class自身声明为final。
  • 将所有成员变量定义为private和final，并且不提供setter方法。
  • 构造对象时,成员变量使用深度拷贝来初始化，而不是直接赋值(防御措施，因为无法确定输入对象不被修改)。
  • 确实需要返回内部值或状态的方法，使用copy-on-write原则，创建私有的copy。

finally

从语法和使用实践角度：

finally是Java保证重点代码一定要被执行的一种机制。我们可以使用try-finally或者try-catch-finally来进行类似关闭JDBC连接、保证unlock锁等动作。 需要关闭的连接等资源，更推荐使用Java7中添加的try-with-resource语句

finalize

从语法和使用实践角度：

finalize是基础类java.lang.Object的一个方法，它的设计目的是保证对象在垃圾收集前完成特定资源的回收。finalize机制已经不推荐使用，并且在JDK9开始被标记为deprecated。

      为什么呢？无法保证finalize什么时候执行，执行的是否符合预期。使用不当会影响性能，导致程序死锁、挂起等。 利用上面的提到的 try-with-resources或者try-finally机制，是非常好的回收资源的办法。如果确实需要额外处理，可以考虑 Java 提供的 Cleaner 机制或者其他替代方法。
      finalize的执行时和垃圾收集关联在一起的，一旦实现了非空的finalize()，就会导致相应对象回收呈现数据量的变慢。因为finalize被设计成对象被垃圾收集前调用，这就意味着实现了finalize方法的对象是个”特殊公民”，JVM要对他进行额外的处理。finalize本质上成为了快速回收的阻碍者，可能导致你的对象经过多个垃圾收集周期才能被回收。System.runFinalization()告诉JVM积极一点，是不是就可以了？这是片面的。问题在于，还是不可预测、不能保证的。实践中，因为finalize拖慢垃圾收集，导致大量对象堆积，也是一种典型的导致OOM的原因。 而且，finalize会掩盖资源回收时的出错信息(默认FinalizerThread在所有的强引用被解除后实现内存清理)。参考java.lang.ref.Finalizer(why?java.lang.ref这包后续需要了解)：

      Java平台目前在逐步使用java.lang.ref.Cleaner来替换掉原有的finalize实现。Cleaner的实现利用了虚(幻象)象引用（PhantomReference），这是一种常见的所谓post-mortem清理机制。吸取了finalize里的教训，每个Cleaner的操作都是独立的，它有自己的运行线程，所以可以避免意外死锁等问题。从可预测性的角度来判断，Cleaner或者虚(幻象)引用改善的程度仍然是有限的，如果由于种种原因导致幻象引用堆积，同样会出现问题。所以，Cleaner 适合作为一种最后的保证手段，而不是完全依赖 Cleaner 进行资源回收，不然我们就要再做一遍 finalize 的噩梦了。注意到很多第三方库自己直接利用虚(幻象)引用定制资源收集，比如广泛使用的 MySQL JDBCdriver 之一的 mysql-connector-j，就利用了虚(幻象)引用机制。虚(幻象)引用也可以进行类似链条式依赖关系的动作，比如，进行总量控制的场景，保证只有连接被关闭，相应资源被回收，连接池才能创建新的连接。
